# 共通前提

以下の QA と根拠文書チャンクに対して、**指定された1観点のみ**を判定してください。
各プロンプトは JSON オブジェクトを1つだけ返します。前後に説明文を付けないこと。

**[元チャンク]が複数個並んでいる場合**は、**全てを合わせて1つの根拠資料**として扱い、
QAがそれら全体から導けるかを判定してください(個別チャンクだけで判断しないこと)。

---

## [ANSWERABILITY]

与えられたチャンクのみで質問に正しく答えられるかを 1〜5 で評価。
- 5: チャンクだけで明確かつ一意に答えられる
- 3: 部分的に答えられるが情報不足/曖昧
- 1: チャンクから全く答えられない

出力:
```json
{"answerability": <1-5の整数>, "reason": "<1-2文>"}
```

---

## [LEAKAGE]

質問文（question）の中に**回答そのもの（または回答の核となる固有名詞・数値・ステップ名）**が
そのまま書かれていないかを判定。
ヒント・前提情報・回答が属するカテゴリ名（例: 「温度保護の手順」「定格電圧」）が
含まれているのは pass（正常な質問の定型）である。
**数値・具体的なステップ名・固有のキーワードが質問文に書かれていて、それがそのまま回答になっている場合のみ fail**。

例:
- 質問「温度保護の手順は？」 回答「警告ログ出力、ファン最大回転、95℃でシャットダウン」
  → 質問に回答のステップ名は含まれていない → **pass**
- 質問「定格電圧（DC 24V）はいくらですか？」 回答「DC 24V」
  → 質問に回答数値が書かれている → **fail**
- 質問「製品Xの色は不明ですか？」 回答「不明」
  → 質問に回答語そのものが含まれる → **fail**

出力:
```json
{"leakage": "pass|fail", "reason": "<1-2文>"}
```

---

## [GROUNDING]

answer が rationale（引用テキスト）から直接導けるかを 1〜5 で評価。
- 5: rationale に回答の根拠が明示され、answer は過不足なく対応
- 3: rationale にある程度根拠はあるが推論や補完が必要
- 1: rationale と answer の対応が成立しない / 別情報が混入

出力:
```json
{"grounding": <1-5の整数>, "reason": "<1-2文>"}
```

---

## [DIFFICULTY_MATCH]

質問が要求する**回答難易度 (answer_level)** を実態から判定する。
(検索難易度はチャンク組成から別途自動算出するので、ここでは回答=推論の重さだけを見る)

- Easy: 抜き出し型、推論不要
- Medium: 1-2ステップの推論、比較・統合
- Hard: 3ステップ以上、条件分岐、計算

`answer_level` に実態のレベルを入れる。宣言値とのズレ (difficulty_match) は後段が
この実態レベルから算出するので、ここでは実態の判定だけに集中する。

出力:
```json
{"answer_level": "Easy|Medium|Hard", "reason": "<1-2文>"}
```
